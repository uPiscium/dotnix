# 役割

変更コードを複数の観点で並列レビューし、軽微な問題は即修正し、大きい懸念はIssueに分ける。

## Phase 1: レビュー対象の特定

### デフォルト

1. `git diff main...HEAD --name-only` で変更ファイル一覧を取る
2. `git diff main...HEAD` で差分内容を取る

### 引数指定

- ユーザーがファイルパスやディレクトリを指定した場合は、その範囲を対象にする

## Phase 2: サブエージェント並列起動

以下の観点でレビューサブエージェントを並列に起動する。

| エージェント | 観点 |
| --- | --- |
| `correctness-reviewer` | バグ、型安全性、ロジックエラー、エッジケース |
| `spec-compliance-reviewer` | README や仕様との整合性 |
| `design-reviewer` | YAGNI / KISS / DRY、既存パターンとの一致 |
| `agent-architecture-reviewer` | エージェント設計品質 |

- 各エージェントは読み取り専用にする
- 変更はさせず、検出と報告のみを行う
- 起動時には対象ファイル一覧と差分内容を渡す

## Phase 3: 結果統合と振り分け

報告を統合し、次の2つに振り分ける。

### A. 即修正

- 意図が明確で、判断に迷わない
- 影響範囲が局所的
- 仕様変更や設計変更を伴わない

例:

- typo
- lint 違反
- 未使用 import の削除
- 明らかなバグ修正
- 命名の不統一

### B. Issue 起票

- 要件変更や大きな仕様変更を伴う
- 修正のスコープが今の作業と大きく異なる
- 複数のアプローチがあり判断が必要
- 技術的負債として残すべき

- Issue は `gh issue create` で作成する
- 1 Issue = 1 トピック
- 重要な判断が必要なものには `help wanted` ラベルを付ける

### 判定に迷う場合

- 即修正か Issue か迷う場合は `AskUserQuestion` で確認する

## Phase 4: 実行

1. 即修正する
2. プロジェクト標準の検証コマンドで確認する
   - 例: `just validate`
3. 必要な Issue を起票する

## Phase 5: 報告

```
## レビュー結果

### 即修正した項目
- [file:line] 内容（修正済み）

### 検証結果
- `just validate` の出力

### Issue 起票した項目
- #<number>: タイトル — 理由

### 問題なし
（問題がなかった場合）
```

## 制約

- Phase 2 → 3 → 4 → 5 の順で進める
- サブエージェントの指摘は鵜呑みにしない
- 即修正したものは必ず `just validate` で確認する
- 即修正しなかった指摘は、Issue にするか、`AskUserQuestion` で確認するかのどちらかに振り分ける
- 問題がなければ簡潔に「問題なし」と報告する
