# 役割

予期しない挙動、テスト失敗、ランタイムエラーに対して、根本原因を特定してから対処する。

## 鉄則

- 根本原因調査なしの修正はしない
- 「とりあえず直す」は避ける

## 4フェーズ

### 1. 根本原因調査

- エラーメッセージやスタックトレースを正確に読む
- 最小の再現手順を確立する
- `git log --oneline -20` で最近の変更を確認する
- エラー箇所から入力元へ逆追跡する

### 2. パターン分析

- `Grep` / `Read` で正常に動く同種の実装を探す
- 問題箇所との差分を整理する
- 原因候補を検証しやすい順に並べる

### 3. 仮説検証

- 1回に1つだけ変える
- 変更は最小限にする
- 仮説ごとに支持 / 棄却を記録する

### 4. 修正実装

- 根本原因が特定できたら、再現確認と修正を進める
- 必要なら修正前に再現条件を固定する
- 最後に検証して戻りを確認する

## エスカレーション

- 3回以上失敗したら、アーキテクチャ前提を疑う
- インターフェースや契約を再確認する
- 必要なら `AskUserQuestion` で相談する

## 警告サイン

- 「たぶんこれが原因」と推測だけで進む
- 同じファイルを何度も雑に直す
- 修正のたびに別の箇所が壊れる

## 使うタイミング

- テストが失敗したとき
- 予期しないランタイムエラーが起きたとき
- どこが壊れているか明確でないとき
- バグ報告を受けて修正前に原因を整理したいとき
